Enviar mensaje POST

El flujo básico: usuario escribe → presiona Enter → mensaje va al servidor → server lo guarda en DB → server lo emite por WebSocket a todos en el canal.

// Estructura de un mensaje { id: "snowflake-uuid", content: "Hola mundo", author_id: "user123", channel_id: "canal456", guild_id: "servidor789", timestamp: "2026-03-29T10:00:00Z", edited_at: null, attachments: [], embeds: [], reactions: {}, reply_to: null // id del mensaje al que responde }
Responder un mensaje
  • Guardar reply_to: message_id en el mensaje
  • Al renderizar, buscar el mensaje original por ID
  • Mostrar preview del mensaje citado encima
  • Clic en la cita → hacer scroll hasta el original
Editar mensaje PATCH
  • Solo el autor puede editar
  • Guardar edited_at con el timestamp
  • Mostrar texto "(editado)" junto al mensaje
  • Emitir evento MESSAGE_UPDATE por WS
Eliminar mensaje DELETE
  • Autor o alguien con permiso puede borrar
  • Soft-delete o hard-delete según tu diseño
  • Emitir MESSAGE_DELETE por WebSocket
  • Todos los clientes eliminan el mensaje del DOM
Reacciones 👍
  • Tabla: message_id, user_id, emoji
  • Toggle: si ya existe → eliminar, si no → insertar
  • Mostrar contador agrupado por emoji
  • Emitir REACTION_ADD / REMOVE por WS
Carga de mensajes (scroll infinito) Clave

Discord carga mensajes en "chunks" por cursor, no por página. Cuando llegas arriba, pide los anteriores.

// Query con cursor GET /messages?channel_id=X&before=message_id&limit=50 // Al montar el canal: 1. Cargar los últimos 50 mensajes (ORDER BY timestamp DESC LIMIT 50) 2. Renderizar en orden cronológico (invertir el array) 3. Scroll automático al fondo 4. Cuando el usuario scrollea hacia arriba y llega al primer mensaje: → Pedir 50 mensajes BEFORE del más antiguo cargado → Agregar al principio del listado (sin mover scroll)

Tip: usa IntersectionObserver en el primer mensaje para detectar cuándo cargar más.

Crear servidor

  • Cambiar icono (subir imagen → guardar URL)
  • Cambiar banner, descripción

  • Permisos de roles para ese canal
  • Emitir CHANNEL_CREATE
Reordenar canales
  • Drag & drop en el sidebar
  • Cada canal tiene campo position
  • Al soltar → actualizar posiciones en batch
  • Emitir CHANNEL_UPDATE a todos
Permisos por canal (overwrites)

Cada canal puede tener permisos que sobreescriben los del rol. Se guarda como una tabla con allow/deny bit flags.

{ channel_id: "...", target_id: "role_id o user_id", type: "role" | "member", allow: 0x0000000400, // bits de permisos permitidos deny: 0x0000000800 // bits de permisos denegados }
Estado "último mensaje leído" (unread)

Tabla read_states con user_id, channel_id, last_read_message_id. Si hay mensajes más nuevos → mostrar punto de no leído. Al abrir el canal → actualizar el last_read.

Perfil de usuario
{ id: "snowflake", username: "juan_dev", discriminator: "0001", email: "juan@email.com", avatar: "url-del-avatar", banner: null, bio: "Soy developer 👋", status: "online" | "idle" | "dnd" | "invisible" | "offline", custom_status: "Codificando 🚀", created_at: "..." }
Estados de presencia
🟢 Online (visible)

El usuario está activo. Aparece en la lista de miembros online. Se emite PRESENCE_UPDATE al conectarse.

🌙 Ausente (idle)

Sin actividad por N minutos. Se puede set manual o automáticamente al detectar inactividad del mouse/teclado.

⛔ No molestar (DND)

Visible pero no recibe notificaciones. El usuario lo activa manualmente.

⚫ Invisible

Aparece como offline para todos, pero puede leer y escribir normalmente. La DB guarda status: "invisible"; a otros se les muestra como offline.

Un canal especial entre 2 usuarios, sin servidor. En la DB es un canal de tipo DM con los dos participantes.

// Abrir DM POST /users/@me/channels { recipient_id: "user_id" } // Si ya existe ese DM → devolver el existente // Si no → crear canal tipo DM y devolver
Grupo de DM (hasta 10 personas)
  • Canal tipo GROUP_DM con múltiples participantes
  • Tiene nombre e icono propios (editables por cualquier miembro)
  • Cualquier miembro puede agregar personas o salir
  • El que crea es "dueño" del grupo y puede cambiar nombre/icono
  • No tiene roles, no tiene permisos complejos

Embeds (preview de links)

Cuando alguien pega una URL, el backend hace un scraping para obtener og:title, og:description, og:image y los muestra como tarjeta dentro del mensaje. El usuario puede cerrar el embed.

Estructura de un rol
{ id: "snowflake", guild_id: "...", name: "Moderador", color: "#e74c3c", // color del nombre en chat hoist: true, // mostrar separado en la lista de miembros position: 3, // jerarquía (más alto = más poder) permissions: bigint, // bitmask de permisos mentionable: true }
Sistema de permisos con bitmask
// Permisos básicos que necesitas implementar: SEND_MESSAGES = 1 << 11 // 0x800 READ_MESSAGES = 1 << 10 MANAGE_MESSAGES = 1 << 13 // borrar mensajes de otros KICK_MEMBERS = 1 << 1 BAN_MEMBERS = 1 << 2 MANAGE_CHANNELS = 1 << 4 MANAGE_GUILD = 1 << 5 MANAGE_ROLES = 1 << 28 ADMINISTRATOR = 1 << 3 // bypass todo // Verificar si tiene permiso: const hasPermission = (userPerms, perm) => (userPerms & perm) === perm;
Asignar roles
  • Solo quien tenga MANAGE_ROLES
  • Solo puede asignar roles de menor jerarquía
  • Guardar en guild_members.roles[]
  • Emitir MEMBER_ROLE_UPDATE por WS
Calcular permisos finales
  • Obtener todos los roles del usuario
  • Hacer OR de todos los permisos
  • Aplicar channel overwrites (allow/deny)
  • Si tiene ADMINISTRATOR → permisos totales
Tema oscuro / claro
  • Guardar preferencia en localStorage o en la DB del usuario
  • Usar CSS variables para todos los colores
  • Clase .theme-dark o .theme-light en el root
  • Toggle instantáneo sin recargar la página

Agrupación de mensajes

Si el mismo usuario envía mensajes consecutivos en menos de 5 minutos, se agrupan: solo se muestra el avatar y nombre en el primer mensaje, los siguientes solo muestran el texto.

// Al renderizar mensajes, comparar con el anterior: const isGrouped = prev && prev.author_id === msg.author_id && (msg.timestamp - prev.timestamp) < 5 * 60 * 1000;

WebSocket — eventos principales WS
// Eventos del SERVIDOR → cliente: MESSAGE_CREATE // nuevo mensaje MESSAGE_UPDATE // mensaje editado MESSAGE_DELETE // mensaje eliminado REACTION_ADD // reacción añadida REACTION_REMOVE // reacción quitada TYPING_START // alguien está escribiendo PRESENCE_UPDATE // cambio de estado (online/offline) CHANNEL_CREATE // nuevo canal creado CHANNEL_UPDATE // canal editado CHANNEL_DELETE // canal eliminado GUILD_MEMBER_ADD // alguien entró al servidor GUILD_MEMBER_REMOVE // alguien salió / fue kickeado GUILD_UPDATE // servidor editado (nombre, icono...) // Eventos del CLIENTE → servidor: IDENTIFY // autenticación inicial HEARTBEAT // "sigo vivo" cada ~45s TYPING_START // el usuario está escribiendo PRESENCE_UPDATE // cambiar tu estado
Ciclo de vida de la conexión WS
1. Cliente abre WebSocket 2. Servidor envía HELLO (con heartbeat_interval) 3. Cliente envía IDENTIFY (con token) 4. Servidor envía READY (con info del usuario, servidores, etc.) 5. Cliente empieza a escuchar eventos 6. Cliente envía HEARTBEAT cada N ms 7. Servidor responde HEARTBEAT_ACK // Si no llega ACK → reconectar
Rooms por canal

Cada canal es una "sala" en Socket.io o similar. Al cambiar de canal, el cliente hace join a la nueva sala. Así los mensajes solo llegan a quien está suscrito a ese canal.

Online/Offline

Al conectar WS → status: online. Al desconectar → status: offline. Emitir PRESENCE_UPDATE en ambos casos para que todos los miembros del servidor actualicen su lista.

Stack recomendado para empezar
Frontend: React + Vite (o Next.js) Backend: Node.js + Express WebSocket: Socket.io (más fácil) o ws puro DB: PostgreSQL (relacional, ideal para permisos) Auth: JWT tokens (access + refresh) Storage: Cloudflare R2 / Supabase Storage / local Deploy: Railway / Render / Fly.io
